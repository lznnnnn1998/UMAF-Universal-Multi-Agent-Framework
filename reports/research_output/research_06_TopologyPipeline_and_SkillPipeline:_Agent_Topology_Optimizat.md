# TopologyPipeline and SkillPipeline: Agent Topology Optimization and Artifact-Agnostic Skill Detection

## Overview

The TopologyPipeline and SkillPipeline represent two of UMAF's most architecturally distinctive pipelines, each embodying a fundamentally different workflow pattern that demonstrates the framework's versatility beyond simple code generation. 

The **TopologyPipeline** (v1.5) implements a meta-cognitive capability: it analyzes an arbitrary multi-agent task description, proposes candidate agent topologies using four canonical design patterns, evaluates them across five dimensions, and produces both a machine-readable specification (`topology_spec.json`) and a human-readable report (`topology_report.md`). This pipeline is essentially a *system that designs systems* — it applies LLM reasoning to the problem of agent graph architecture, treating agent topology selection as an optimization problem over a space of LangGraph `StateGraph` configurations. The pipeline follows a strict linear 4-node graph (analyzer → designer → evaluator → writer) with no retry loops, reflecting the nature of the task: each stage transforms the output of the previous stage without needing to revisit earlier decisions. This "waterfall" design contrasts with UMAF's iterative coder↔reviewer loops and represents a different point in the reliability-vs-refinement tradeoff space.

The **SkillPipeline** (v1.5, v2 detectors) implements a fundamentally different pattern: a fan-out/fan-in graph where a single scanner classifies an artifact, then four parallel detectors analyze it across independent dimensions, followed by an aggregator that deduplicates and categorizes, and a writer that produces structured output. This pipeline analyzes any project — codebase, research paper, blog article, dataset, design document, presentation, or configuration repository — and infers the *human skills* demonstrated by its creator(s) through evidence in the artifact. The key architectural innovation is the artifact-agnostic detection: the four detectors (DomainExpertise, TechnicalCraft, Methodology, Rigor) read the same `artifact_analysis.json` but extract completely different dimensions of skill, enabling truly orthogonal analysis without detector-to-detector coupling. This is a practical implementation of the "multiple perspectives on the same evidence" pattern from ensemble learning, adapted for LLM-based analysis.

Both pipelines were originally generated by UMAF's own CoderPP pipeline from markdown specification files — a recursive self-application that validates the framework's code generation capabilities. The TopologyPipeline optimizing topologies and the SkillPipeline detecting skills are together UMAF's meta-analysis layer: they analyze either the *structure* of agent systems (how should agents be organized?) or the *content* of artifacts (what skills does this demonstrate?).

## Key Methods & Approaches

### 1. TopologyPipeline: The 4-Node Linear Graph

The TopologyPipeline (`pipeline/topology.py`) implements a strict linear workflow with four nodes, each backed by an `AgentRole` subclass. The pipeline has no cycles, no retry loops, and no parallel execution — it is the simplest graph topology in UMAF, chosen deliberately for a workflow where each stage has a single input and a single output, and no iteration is needed.

**State Management**: `TopologyState` (`topology.py:16-25`) is a TypedDict with eight fields tracking the complete data flow:
- `input_spec`: The original task description
- `complexity_factors`: Dict from the analyzer containing six factor assessments
- `candidate_topologies`: List of 2-4 topology proposals from the designer
- `evaluated_topologies`: Scored and ranked topologies from the evaluator
- `topology_spec`: Final output from the writer (containing spec, report paths, and success flag)
- `status`: Flow routing token driving the LangGraph conditional edges

**Flow Routing** (`topology.py:141-148`): The status-based router uses a simple forward-only flow map:
```
initialized → analyzer → analyzed → designer → designed → evaluator → evaluated → writer → written → END
```
Terminal error statuses (`error_analysis_failed`, `error_design_failed`, `error_evaluation_failed`, `error_writer_failed`) provide clean failure exits at each stage. Each node has a try/except wrapper that catches exceptions and sets the appropriate error status, preventing pipeline crashes.

**Key Design Decision — No Decomposition**: Unlike ResearchPipeline or CoderPPPipeline, which decompose complex tasks into sub-tasks for parallel execution, TopologyPipeline's `_decompose()` returns an empty list (`topology.py:39`). The pipeline graph itself handles all work — there are no independent workers, so no decomposition is needed. This is because topology optimization is inherently sequential: you cannot evaluate topologies before designing them, and you cannot write a final spec before evaluation. The task does not benefit from parallelism at the sub-task level.

### 2. TopologyAnalyzerRole: The 6-Factor Complexity Assessment

`TopologyAnalyzerRole` (`topology/analyzer.py:12-181`) is the pipeline's entry point. With `agent_name = "topology_analyzer"` and `max_steps = 8`, it has the shortest step budget of the topology roles — the analysis task is well-scoped and the output format is strictly structured, so the agent should not need extensive exploration.

**The 6 Complexity Factors**: The `build_task()` method (`analyzer.py:33-96`) constructs a prompt that instructs the LLM to assess the input task across six orthogonal dimensions, each assigned a level (low/medium/high) with 1-2 sentences of reasoning:

1. **data_dependencies** — How much data must flow between agents? This determines whether a sequential or fan-out topology is appropriate. High data dependency favors sequential topologies where each stage can consume the previous stage's output directly.

2. **parallelism_opportunities** — Can sub-tasks be executed independently? This is the primary driver of the fan-out/fan-in pattern. High parallelism opportunities make the debate_consensus pattern viable (independent agents solving the same problem).

3. **tool_requirements** — How many distinct tools does each agent need? Are there conflicting or restricted tool needs? This affects agent role design: agents with conflicting tool requirements may need to be separated into different pipeline stages.

4. **error_domains** — How many distinct failure modes exist? What is the blast radius of a single agent failure? High error domain count favors hierarchical topologies where domain leads can contain failures within their sub-domain, preventing cascading failures.

5. **latency_sensitivity** — Is end-to-end latency critical (real-time/interactive) or relaxed (batch processing)? Latency-sensitive tasks favor fan-out parallelism over sequential execution. Batch-tolerant tasks can use sequential topologies for simplicity.

6. **scale** — How many sub-tasks/agents are needed? Is the workload compute-bound or I/O-bound? This determines whether the topology needs to support many parallel workers (fan-out) or a deep hierarchy with multiple levels of delegation.

**Output Parsing**: `parse_result()` (`analyzer.py:100-134`) implements a three-tier extraction strategy common across UMAF's roles:
1. **Agent messages**: Iterates reversed messages, extracts JSON objects via `extract_json_object()` from `utils.py`, validates the presence of a `"factors"` key
2. **Disk file**: Falls back to reading `complexity_analysis.json` from the working directory
3. **Fallback generation**: Calls `_fallback_analyze()` which uses word count heuristics to estimate scale and assigns conservative "medium" values to all other factors

**Fallback Behavior**: `_fallback_analyze()` (`analyzer.py:139-181`) is a deterministic heuristic that uses input spec word count to estimate scale: `word_count > 100 → "high"`, `> 30 → "medium"`, otherwise `"low"`. All other factors default to "medium" with generic reasoning — a deliberately conservative strategy that errs on the side of caution rather than making unfounded assumptions.

### 3. TopologyDesignerRole: The 4-Pattern Catalog

`TopologyDesignerRole` (`topology/designer.py:12-319`) proposes 2-4 candidate topologies. With `max_steps = 12`, it has the highest step budget in the pipeline — design is the most creative and open-ended stage.

**The 4 Design Patterns**: The `_PATTERNS` class attribute (`designer.py:24-42`) defines a catalog of four canonical multi-agent topologies, each with a description optimized for the LLM prompt:

1. **sequential**: "Agents execute one after another in a fixed order. Best for: strictly ordered workflows with no parallelism opportunities." This is the simplest pattern — a linear chain where each agent's output feeds the next. It minimizes coordination overhead at the cost of total latency being the sum of all agent times.

2. **fan_out_fan_in**: "A dispatcher fans out work to parallel workers, then a collector aggregates results. Best for: embarrassingly parallel sub-tasks." This is the pattern used by UMAF's own ResearchPipeline and SkillPipeline. It trades coordination complexity for reduced wall-clock time through parallelism.

3. **debate_consensus**: "Multiple agents independently solve the same problem, then a judge selects or synthesizes the best answer. Best for: high-stakes decisions where correctness matters more than speed." This pattern prioritizes reliability over cost and latency — it is the most expensive pattern per-run but provides the strongest correctness guarantees through redundancy.

4. **hierarchical**: "A top-level orchestrator delegates to sub-orchestrators, which in turn manage worker agents. Best for: complex, multi-layered tasks." This pattern scales to the largest agent counts by introducing intermediate coordination layers. The two-level parallelism (domains run concurrently, workers within domains run concurrently) maximizes throughput for complex tasks.

**Topology Specification Schema**: Each candidate topology must specify:
- `name`, `pattern`, `description` — identification and rationale
- `agents` — list of agent configurations with `agent_name`, `role_type` (description of function), `tools` (list of tool names), and `max_steps`
- `connections` — list of `{from, to}` edges defining the communication graph
- `parallelism_strategy` — description of how parallel execution is used (or "none")
- `strengths` and `weaknesses` — 2-3 advantages and limitations each

**Output Parsing**: Uses `extract_json_array()` from `utils.py` (rather than `extract_json_object()`) since the output is a JSON array of topology objects. Falls back to reading `candidate_topologies.json` from disk, then to `_fallback_design()` which generates three default topologies (sequential, fan-out/fan-in, hierarchical) with pre-configured agents and connections.

**Fallback Design**: The `_fallback_design()` method (`designer.py:206-319`) generates three concrete topologies:
- **Sequential Pipeline**: 3 agents (analyzer → worker → writer), 2 connections, no parallelism
- **Fan-Out/Fan-In Pipeline**: 4 agents (head → two workers → aggregator), 4 connections, parallel workers
- **Hierarchical Orchestrator**: 6 agents (orchestrator → two domain leads → two workers → synthesizer), 6 connections, two-level parallelism

Each fallback topology includes realistic tool assignments and thoughtful strengths/weaknesses analysis, making them useful even as the primary output if LLM generation fails.

### 4. TopologyEvaluatorRole: The 5-Dimension Scoring Rubric

`TopologyEvaluatorRole` (`topology/evaluator.py:12-269`) is the most analytically rigorous stage. With `max_steps = 10`, it must read the candidate topologies, score each across five dimensions, compute totals, and produce a ranked list.

**The 5 Evaluation Dimensions**: Each dimension is scored 1-10 with reasoning. The `_DIMENSIONS` list and `_dimension_guide()` method (`evaluator.py:22-28, 154-178`) define:

1. **latency** — End-to-end execution time. Sequential topologies score low (all times sum); fan-out topologies score high (parallel execution minimizes critical path). The guide emphasizes that "critical path length matters" — not just total agent count.

2. **reliability** — Fault tolerance and error recovery. Debate_consensus scores highest (redundancy); sequential scores lowest (single point of failure at each stage). Independent workers that isolate failures improve scores, as do explicit retry loops.

3. **cost_efficiency** — Total tokens and API calls per run. Sequential scores highest (fewest agents, no redundancy); debate_consensus scores lowest (multiple agents solving the same problem). "Fewer agents and simpler prompts reduce cost."

4. **simplicity** — Ease of implementation and maintenance. Sequential scores highest (trivial to implement in LangGraph); hierarchical scores lowest (complex DAG with two-level coordination). "Linear flows are simpler than complex DAGs."

5. **scalability** — Ability to handle growing task size or sub-task count. Hierarchical scores highest (add domains or workers per domain); sequential scores lowest (fixed throughput regardless of task size).

**Heuristic Fallback Scoring**: `_fallback_evaluate()` (`evaluator.py:203-269`) implements per-pattern heuristic scoring based on the topology's structural characteristics. The scores reflect deliberate tradeoffs encoded in the evaluator's design philosophy:

| Pattern | Latency | Reliability | Cost Eff. | Simplicity | Scalability | Total |
|---------|---------|-------------|-----------|------------|-------------|-------|
| sequential | 4 | 5 | 8 | 10 | 3 | 30 |
| fan_out_fan_in | 8 | 7 | 6 | 7 | 8 | 36 |
| debate_consensus | 3 | 10 | 3 | 5 | 5 | 26 |
| hierarchical | 7 | 6 | 4 | 4 | 9 | 30 |

The fan-out/fan-in pattern scores highest overall (36/50) in the fallback heuristic, reflecting UMAF's own design bias — it is the pattern used by three of the seven pipelines. The debate_consensus pattern, despite having the highest reliability, scores lowest overall due to poor cost efficiency and latency — a tradeoff that is explicitly documented in the scoring guidelines.

**Sorting**: After evaluation (whether LLM or fallback), results are sorted by `total_score` descending (`evaluator.py:148`), ensuring the writer always receives a ranked list.

### 5. TopologyWriterRole: Dual-Format Output Generation

`TopologyWriterRole` (`topology/writer.py:11-395`) is the most restricted role in all of UMAF — it has only the `write_file` tool (`tools_config.json:160-162`), no `read_file`. With `max_steps = 8`, it must produce two files: a machine-readable JSON spec and a human-readable markdown report. The role is intentionally restricted because the writer receives all necessary data via the `evaluated_topologies` and `candidate_topologies` parameters — it should not need to discover files from disk.

**Output 1 — `topology_spec.json`**: Contains:
- `recommended_topology`: Name of the highest-scoring topology
- `total_score` and `max_possible_score` (50)
- `design_pattern`, `agents`, `connections`, `parallelism_strategy` — copied from the candidate topology
- `pipeline_implementation_guide`: An implementation-ready guide with:
  - `overview`: High-level pipeline description
  - `nodes`: List of agent nodes with role types, descriptions, and transition targets
  - `flow_diagram`: ASCII art showing the node graph (generated by `_build_flow_diagram()`)
  - `key_design_decisions` and `configuration_notes`

**Output 2 — `topology_report.md`**: Contains a comparison table of all candidate topologies with per-dimension scores, a recommendation section explaining the selection rationale, implementation notes, and suggested next steps.

**ASCII Flow Diagram Generation**: `_build_flow_diagram()` (`writer.py:273-321`) constructs pattern-specific ASCII diagrams:
- Sequential: `[agent1] → [agent2] → [agent3]`
- Fan-out: `[head]` with branches to `[worker_1] [worker_2] [worker_3]` converging on `[collector]`
- Debate: Three agents with arrows converging on `[judge/synthesizer]`
- Hierarchical: Two-level tree with orchestrator → domain leads → workers

**Fallback Generation**: If the LLM fails to produce either file, `_generate_spec()` and `_generate_report()` provide complete programmatic fallbacks with all required sections. The fallback report includes a comparison table, recommendation, implementation notes, and next steps, ensuring the pipeline always produces usable output.

### 6. SkillPipeline: The Fan-Out/Fan-In Graph

The SkillPipeline (`pipeline/skill.py`) implements a 4-node fan-out/fan-in graph where the "detectors" node internally runs four agent roles in parallel. This graph topology directly mirrors the fan-out/fan-in pattern that the TopologyPipeline's own evaluator scores highest.

**State Management**: `SkillState` (`skill.py:78-87`) tracks eight fields:
- `input_spec` and `project_dir`: The target to analyze (directory path or requirement text)
- `project_scan`: Surface-level file enumeration from the scanner
- `artifact_analysis`: Deep analysis including artifact type, content samples, and structure
- `detector_outputs`: List of four per-detector result dicts (output from the fan-out)
- `skill_inventory`: Aggregated skills and tools from the aggregator (input to the writer)
- `status`: Flow routing token

**Flow Routing** (`skill.py:260-266`):
```
initialized → scanner → scanned → detectors → detected → aggregator → aggregated → writer → written → END
```
The router also accepts `detected_partial → aggregator` — following UMAF's "always move forward" principle, partial detector results are still routed for aggregation rather than aborting the pipeline. Terminal errors include `error_no_detector_outputs` and `error_no_inventory`.

**Parallelism Limitation for Claude CLI** (`skill.py:170`): When the backend is `claude_cli`, `max_workers` is set to `1` instead of `4`. This prevents out-of-memory conditions from four simultaneous heavy `claude -p` subprocesses — a pragmatic engineering constraint documented in the code. For the DeepSeek backend, all four detectors run in true parallel via `ThreadPoolExecutor`.

**Dependency Injection Fix (v1.6.1)**: The detector node receives `artifact_analysis` and `project_scan` directly from state. The aggregator receives `detector_outputs` from state, with inline summaries embedded in its prompt via `build_task()` so it knows what was found without having to discover files from disk. This pattern — passing upstream outputs as `execute()` kwargs with prompt-embedded summaries — was the fix for the v1.6.1 bug where downstream agents were blind to upstream computation.

### 7. SkillScannerRole: Artifact Classification with 8 Categories

`SkillScannerRole` (`skill/scanner.py:320-733`) is the pipeline's most complex role, with `max_steps = 15` — the highest step budget of any skill role. It performs four distinct phases in a single agent invocation:

**Phase 1 — Surface Scan**: Runs `find` and `ls -laR` commands to enumerate all files (excluding `.git`, `__pycache__`, `node_modules`, `.venv`, `venv`, `dist`, `build`, `.tox`). Produces `project_scan.json` with file categories (source, config, docs, test, build, ci, other), key file identification, and directory tree preview.

**Phase 2 — Artifact Classification**: Classifies the project into one of eight types defined in `_ARTIFACT_SIGNATURES` (`scanner.py:20-91`):
1. **software_project**: Source code with build configs — indicators include source extensions (`.py`, `.js`, `.go`, etc.), build files (`setup.py`, `package.json`, `Cargo.toml`), and a minimum 15% source file ratio
2. **research_paper**: LaTeX/PDF with academic structure — indicators include `.tex`/`.pdf`/`.bib` extensions and section headers (Abstract, Introduction, Methodology, etc.)
3. **blog_article**: Markdown posts with frontmatter — indicators include `.md`/`.mdx` extensions, frontmatter keys (`title:`, `date:`, `tags:`), and platform indicators (`_posts/`, `blog/`)
4. **documentation**: Structured docs with API references — indicators include doc tools (`mkdocs.yml`, `conf.py`, `docusaurus.config.js`) and a high doc file ratio (40%)
5. **dataset**: Data files with schemas — indicators include data extensions (`.csv`, `.jsonl`, `.parquet`, `.sqlite`) and schema files, with a minimum 30% data file ratio
6. **design_document**: Architecture specs with diagrams — indicators include keywords (architecture, design, specification, RFC) and diagram extensions (`.drawio`, `.fig`)
7. **presentation**: Slide decks — indicators include `.pptx`/`.key` extensions and tools (reveal.js, Marp, Beamer)
8. **configuration**: Dotfiles and infrastructure-as-code — indicators include a high config ratio (60%) and config directories (`.config`, `terraform`, `kubernetes`)

**Deterministic Classification Algorithm**: `_classify_artifact()` (`scanner.py:94-200`) implements a scoring-based classification that works without an LLM. For each artifact type, it computes a weighted score from multiple evidence sources:
- Source file ratio (weight: 3.0 if threshold met)
- Extension matches (weight: 0.5 per match)
- Build file matches (weight: 1.5 per match)
- Doc tool matches (weight: 2.0 per match)
- Config ratio (weight: 3.0 if threshold met)
- Keyword matches in content (weight: 0.5 per match)
- Section header matches (weight: 1.0 per match)
- Frontmatter key matches (weight: 0.5 per match)

Confidence is assigned based on the best score: `≥5 → "high"`, `≥2.5 → "medium"`, otherwise `"low"`.

**Phase 3 — Deep Reading**: `_sample_content()` (`scanner.py:203-272`) reads up to 12 representative files (max 2000 chars each) in three passes:
1. Priority files: README, CONTRIBUTING, CHANGELOG, LICENSE, build configs
2. Source files from different directories (one per directory)
3. Remaining files with unseen extensions (one per extension)

This three-pass strategy ensures broad coverage while respecting the 12-file budget.

**Phase 4 — Structure Analysis**: `_analyze_structure()` (`scanner.py:275-316`) maps the artifact's organization: top-level components, directory depth, total directories, and sub-components (directories with 2+ files). This structural information helps downstream detectors understand how the artifact is organized.

**Fallback Surface Scan**: `_fallback_surface_scan()` (`scanner.py:574-726`) runs actual `find` and `ls` shell commands, categorizes files by extension and naming conventions, and identifies test files (matching patterns like `test_*`, `*_test.py`, `/test/`, `/tests/`, `__tests__/`), CI files (GitHub Actions, GitLab CI, Jenkins), and documentation files. This is a fully functional fallback that can operate without any LLM invocation, producing a complete `project_scan.json` with file categories, key files, and directory tree preview.

### 8. The Four Artifact-Agnostic Detectors: Parallel Skill Dimension Analysis

The four detector roles in `skill/detectors.py` all inherit from `_BaseDetectorRole` (`detectors.py:78-204`), which provides shared parsing logic, placeholder validation, and fallback patterns. Each detector reads the same `artifact_analysis.json` (loaded via `_load_artifact_analysis()`) but asks a fundamentally different question about it.

**Shared Base Class Capabilities**:

- **Three-tier JSON extraction** (`parse_result()`, lines 88-157): First, extracts JSON from agent AIMessage content using `extract_json_object()`. Second, for claude_cli backend, scans `Write` tool-call parameters for embedded report JSON (a critical second pass because Claude CLI agents write output via tool calls rather than raw text). Third, reads from disk. This three-tier strategy handles the different output patterns of the two backends.

- **Template validation** (`_is_valid_report()`, lines 159-200): Rejects reports that are copies of the prompt template. Checks for pipe-delimited schema hints in proficiency/confidence fields (`"advanced|intermediate|beginner"`), angle-bracket placeholders (`<SKILL_NAME>`), and literal `"..."` evidence strings. This prevents the aggregator from receiving template copies masquerading as real results — a common LLM failure mode where the model outputs the schema instead of filling it in.

**Detector 1 — DomainExpertiseDetectorRole** (`detectors.py:292-385`):
- **Question**: "What does the creator know deeply about?"
- **Detection strategy**: Scans content for specialized terminology from nine pre-defined domains: Machine Learning, Distributed Systems, Security, Compiler Design, Database Systems, Game Development, Finance, Scientific Computing, Natural Language Processing
- **Signal catalog**: `_DOMAIN_SIGNALS` (`detectors.py:212-261`) maps each domain to a list of indicator terms (e.g., "neural network", "gradient descent", "transformer" for ML; "Paxos", "Raft", "CAP theorem" for distributed systems)
- **Proficiency heuristics**: ≥5 signal matches → "advanced", ≥2 → "intermediate", ≥1 → "beginner"
- **Confidence**: ≥4 matches → "high", ≥2 → "medium", else "low"
- **Fallback**: `_detect_domain_expertise()` calls domain signal matching, then `_detect_common_tools()` for tool detection from file extensions

**Detector 2 — TechnicalCraftDetectorRole** (`detectors.py:528-615`):
- **Question**: "How skilled is the creator at the medium itself?"
- **Adaptive behavior**: Automatically switches between code craft detection and writing craft detection based on artifact type. Software projects get code-focused analysis (design patterns, error handling, type systems, code organization, performance, security). Non-code artifacts get writing-focused analysis (argumentation, technical writing, narrative structure, clarity).
- **Penalty system**: The error handling dimension includes penalty indicators: `except Exception` and bare `except: pass` reduce the proficiency score, distinguishing between mature error handling and superficial try/catch usage
- **Proficiency thresholds**: Each skill dimension has `threshold_advanced` and `threshold_intermediate` values. Meeting the advanced threshold + 3 additional matches elevates to "expert"
- **Fallback**: `_detect_code_craft()` (for software) or `_detect_writing_craft()` (for non-code artifacts), both using keyword-based indicator matching

**Detector 3 — MethodologyDetectorRole** (`detectors.py:786-874`):
- **Question**: "What tools, workflows, and processes are evident?"
- **Tool detection catalog**: `_TOOL_INDICATORS` (`detectors.py:622-669`) covers 35+ tools across categories: Languages & Runtimes (Python, JS, Go, Rust, Java), Testing (pytest, Jest, unittest), CI/CD (GitHub Actions, GitLab CI, Jenkins), Containers (Docker, Kubernetes), Documentation (LaTeX, Sphinx, MkDocs, Jupyter), Data Science (Pandas, PyTorch, scikit-learn), Web Frameworks (React, Vue, Django, Flask, FastAPI), Build (Webpack, Vite, ESLint), and Infrastructure (Terraform, Ansible)
- **Methodology skill dimensions**: Git Workflow Maturity (conventional commits, branching), CI/CD Sophistication (pipeline complexity), Dependency Management (lock files, version pinning), Environment Management (Docker, venv, containers), Incremental Development (TODOs, changelogs, migrations)
- **File signal weighting**: File matches count double in proficiency calculations — a `pyproject.toml` file is stronger evidence of Python than a text mention of "Python"
- **Fallback**: `_detect_tools()` for deterministic tool detection from file indicators, plus `_detect_methodology_skills()` for process inference

**Detector 4 — RigorDetectorRole** (`detectors.py:996-1083`):
- **Question**: "How thorough, careful, and complete is the work?"
- **Metrics-driven analysis**: Uses quantitative metrics where possible: test file ratio, documentation file ratio, quality config presence
- **Test coverage signals**: Counts test files, computes test_ratio, and searches for testing patterns in content (assert, expect, test(, describe(, parameterize, fixture, mock, stub, edge case, corner case, boundary)
- **Documentation signals**: Searches for documentation content patterns (readme, getting started, installation, usage, API, example, tutorial, guide, reference, FAQ, troubleshooting)
- **Quality enforcement**: Detects linting/formatting tools (ESLint, Prettier, Ruff, mypy, flake8, pylint, pre-commit, editorconfig)
- **Academic rigor** (non-code artifacts): Looks for citations, references, methodology sections, limitations, data availability, reproducibility statements
- **Fallback**: `_detect_rigor_skills()` with separate code and academic rigor paths

### 9. SkillAggregatorRole: Cross-Detector Deduplication and Categorization

`SkillAggregatorRole` (`skill/aggregator.py:37-381`) merges the four detector reports into a unified `skill_inventory.json`. With `max_steps = 10`, it has a moderate step budget — aggregation is primarily a data transformation task, not a creative one.

**Deduplication Strategy**: Tools and skills are deduplicated by name. When the same tool/skill appears in multiple detector reports:
- **Proficiency**: Keep the highest proficiency score (using `_PROFICIENCY_SCORES` from `utils.py`: expert=4, advanced=3, intermediate=2, beginner=1)
- **Sources**: Merge source lists (tracking which detectors found each skill)
- **Evidence**: Merge evidence arrays/list, keeping unique items, capped at 10

**Category Normalization**: Tools are mapped to standard categories via `_CATEGORY_MAP` (`aggregator.py:13-35`):
```
web_framework → "Web Frameworks"
testing → "Testing"
ci_cd → "CI/CD"
containerization → "Containerization"
data_science → "Data Science & ML"
infrastructure_as_code → "Infrastructure as Code"
```

Skills are mapped to five universal categories via `_SKILL_CATEGORY_MAP` (`aggregator.py:264-293`):
- **Domain Knowledge**: Specialized domain expertise (inherited from DomainExpertise detector)
- **Technical Craft**: Design patterns, error handling, type systems, code organization, performance, security, argumentation, writing
- **Engineering Practice**: Git workflow, CI/CD, dependency management, environment management, release management, incremental development
- **Quality Assurance**: Testing strategy, test coverage, documentation quality, code quality enforcement, academic rigor
- **Architecture**: Component design, API design, data modeling, scalability, infrastructure design

**Fallback Aggregator**: `_fallback_aggregator()` (`aggregator.py:203-381`) implements the same deduplication logic deterministically — reading all four detector reports from disk, merging tools and skills with the same name-based deduplication algorithm, computing summary statistics (proficiency distribution, dimensions covered), and reading file stats from `project_scan.json`. The fallback is functionally equivalent to the LLM path, ensuring the pipeline never produces empty output.

**Summary Statistics**: The aggregator computes:
- `total_tools` and `total_inferred_skills` — counts of unique entries
- `dimensions_covered` — which of the four detectors produced results
- `proficiency_distribution` — counts per proficiency level across all tools and skills

### 10. SkillReportWriterRole: Structured JSON + Markdown Output

`SkillReportWriterRole` (`skill/writer.py:45-426`) produces two final output files. With `max_steps = 8`, it is primarily a formatting and organization task.

**`skills.json`** — Structured machine-readable output with:
- `project`, `artifact_type`, `generated_at` — metadata
- `summary` — total tools, total skills, dimensions, proficiency distribution
- `tools_by_category` — tools grouped by normalized category
- `skills_by_category` — inferred skills grouped by universal category
- `all_tools` and `all_skills` — flat lists with full metadata (name, category, proficiency, confidence, sources)

**`skills_report.md`** — Human-readable output with:
- Title with project name and generation date
- Executive summary with key stats
- Proficiency distribution table with visual bars (using `█` characters)
- Detected tools table (up to 30 tools with proficiency badges)
- Inferred skills by category sections with per-skill proficiency and confidence
- File statistics table
- Recommendations section (identifying gaps: no expert-level skills, too many beginners, no tests, etc.)

**Proficiency Badge System** (`writer.py:13-18`):
```
expert → 🟣 Expert
advanced → 🟢 Advanced
intermediate → 🟡 Intermediate
beginner → ⚪ Beginner
```

### 11. Comparison with Related Work

**Topology Optimization in Multi-Agent Systems**: UMAF's TopologyPipeline occupies a unique position in the landscape of automated topology design. Systems like G-Designer (Zhang et al., 2024) use graph neural networks with variational auto-encoders to generate task-adaptive communication topologies, achieving 95.33% token reduction on benchmarks. ARG-Designer (2025, AAAI 2026 oral) uses autoregressive graph generation to jointly determine agent count, roles, and communication links. MASS (Feb 2025) interleaves prompt and topology optimization in three stages. In contrast, UMAF's approach is LLM-native rather than learned — the topology is proposed by an LLM reasoning about task characteristics using structured prompts rather than by a trained graph generator. This trades the efficiency and benchmark performance of learned approaches for interpretability (every topology comes with natural language rationale) and zero-shot generalization (no training data needed for new task types). It also means the pipeline can explain *why* it chose a particular topology, a capability that learned graph generators currently lack.

**Skill Detection from Software Artifacts**: The problem of inferring developer skills from code repositories has been approached primarily through two paradigms: (1) GitHub profile analysis using commit patterns, language distributions, and contribution graphs, and (2) repository structure analysis using file organization, test coverage, and tool configuration. UMAF's SkillPipeline is unusual in combining both approaches within an LLM-based framework: the scanner provides the quantitative foundation (file counts, extension distributions, test ratios), while the four detectors add semantic understanding (domain terminology recognition, code pattern analysis, methodology inference). The artifact-agnostic design — where the same four detectors work on software projects, research papers, blog articles, and datasets — is particularly novel. Most existing systems (e.g., JD-driven agentic GitHub evaluators) are tightly coupled to specific artifact types (usually code repositories).

**Fan-Out/Fan-In Pattern in LLM Pipelines**: The SkillPipeline's detector parallelism implements the fan-out/fan-in pattern that the TopologyPipeline itself recommends as the highest-scoring general-purpose topology. This creates an interesting meta-circularity: the pipeline that recommends topologies uses the pattern that its own evaluator scores highest. The 4-detector parallelism is also a practical demonstration of the tradeoffs the evaluator identifies: parallelism reduces wall-clock time (latency score: 8) at the cost of increased coordination complexity (simplicity score: 7) and moderate cost overhead (cost efficiency: 6). For the claude_cli backend, the forced sequential execution (max_workers=1) due to memory constraints is a concrete example of the evaluator's "scalability" dimension — the topology can handle parallel execution, but infrastructure constraints may limit it in practice.

## Open Questions & Future Directions

1. **Closing the topology recommendation-to-execution loop**: The TopologyPipeline currently *recommends* an optimal topology but does not *execute* it. The next logical step would be a meta-executor that takes the `topology_spec.json` output, dynamically constructs a LangGraph `StateGraph` from the specification, instantiates the specified agents, and runs the pipeline. This would close the loop from analysis to execution, making the TopologyPipeline not just a recommender but a topology executor.

2. **Learned vs. LLM-native topology optimization**: G-Designer and ARG-Designer demonstrate that learned graph generators can produce optimized topologies with lower token cost than LLM reasoning. However, they sacrifice interpretability. A hybrid approach — using learned generators to propose candidate topologies and LLMs to evaluate and explain them — could combine the strengths of both paradigms.

3. **Joint prompt-topology optimization**: MASS demonstrates that prompts and topologies should be optimized jointly, not independently. Currently, UMAF's TopologyPipeline only optimizes topology; agent prompts are fixed in `AgentRole.build_task()`. Extending the pipeline to also propose optimized prompts for each agent role would address this coupling.

4. **Dynamic skill detection across project versions**: The SkillPipeline analyzes a single snapshot of a project. Analyzing skill evolution across git history — detecting when a skill was acquired, how proficiency changed over time, and which commits demonstrated skill growth — would provide a temporal dimension to skill analysis.

5. **Cross-detector conflict resolution**: The current aggregator uses a simple "keep highest proficiency" strategy when detectors disagree. A more sophisticated approach could use detector confidence scores to weight proficiency assessments, or flag conflicts for human review when detectors produce contradictory skill assessments.

6. **Skill ontology standardization**: The current skill categories (Domain Knowledge, Technical Craft, Engineering Practice, Quality Assurance, Architecture) are UMAF-specific. Mapping to standard frameworks like the European e-Competence Framework (e-CF) or SFIA (Skills Framework for the Information Age) would make SkillPipeline output interoperable with HR and professional development systems.

7. **Topology evaluation with real execution data**: The TopologyEvaluator currently scores topologies based on LLM reasoning about their characteristics. Running candidate topologies on actual tasks and measuring real latency, cost, and reliability would provide ground truth scores. This could create a feedback loop where measured performance improves future evaluations.

8. **Adaptive detector selection**: The SkillPipeline always runs all four detectors, even when some are unlikely to produce useful results (e.g., Rigor detection on a single-file script). A pre-classification step that selects which detectors to run based on artifact type and size could reduce cost for simple artifacts.

9. **Multi-modal artifact analysis**: The current scanner reads text content only. Extending the scanner to analyze images (diagrams in design documents, plots in notebooks), audio, or video would expand the SkillPipeline's applicability to multimedia artifacts.

10. **Topology pattern library expansion**: The four patterns (sequential, fan_out_fan_in, debate_consensus, hierarchical) cover common topologies but miss patterns like event-driven (publish/subscribe), streaming (continuous data flow), and recursive (self-similar decomposition). Adding these patterns to the designer's catalog would increase the space of possible recommendations.

## Relevance to Main Topic

The TopologyPipeline and SkillPipeline represent UMAF's meta-analysis layer — they analyze either the structure of agent systems (how should agents be organized for optimal performance?) or the content of artifacts (what skills does this artifact demonstrate?). Together they demonstrate that UMAF is not just a code generation framework but a general-purpose multi-agent analysis platform.

The TopologyPipeline is particularly significant because it applies UMAF's agent capabilities to the problem of UMAF's own architecture. It is a meta-system — a system that designs agent systems — and represents a step toward self-optimizing agent frameworks. The fact that it was generated by UMAF's own CoderPP pipeline (a system designing a system that designs systems) demonstrates the recursive self-application that the SelfEvolutionPipeline formalizes in v1.8.

The SkillPipeline demonstrates that UMAF's architecture generalizes beyond code generation to analytical tasks. The artifact-agnostic detector design — where four independent detectors analyze the same evidence through different lenses — is a pattern that could be applied to any multi-perspective analysis task (code review from security/performance/maintainability perspectives, document analysis from clarity/completeness/accuracy perspectives, etc.). The fan-out/fan-in pattern that the SkillPipeline implements is recommended by the TopologyPipeline as the optimal general-purpose topology, creating a closed validation loop where the framework's own analysis tools confirm its architectural choices.

Both pipelines exemplify UMAF's design philosophy of structured LLM reasoning with deterministic fallbacks. Every stage in both pipelines has a complete fallback implementation that produces valid output without any LLM invocation. This ensures pipeline reliability — the outputs may be less nuanced without LLM reasoning, but they will always be structurally complete and usable. The comprehensive test coverage (14 tests for TopologyPipeline, 15 tests for SkillPipeline, all passing in the 379-test suite) validates both the LLM paths and the fallback chains end-to-end, serving as a regression safety net for future development.
